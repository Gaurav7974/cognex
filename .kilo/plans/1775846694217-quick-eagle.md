# CHP Primitive Proof: ΔC Minimal Validation

## Goal
Strip CHP to its minimal form and prove ΔC (Delta Cognition) as the universal primitive for cognition exchange. Validate that a sequence of ΔC changes enables reliable task continuation across agents and models, establishing the foundation before building the full system.

## Minimal ΔC Primitive
- **Primitive**: ΔC as controlled symbolic assignment (field → value), not raw cognition primitive—pattern: "change [field] to [value]", fixed field dictionary per task (predefined, identical across agents, e.g., auth_method, framework), normalized field names, controlled value sets or normalization rules per field.
- **State Model**: Simple list of changes, with effective state = last value per normalized field derived from sequence; unset fields = explicitly undefined (no implicit defaults).
- **Replay Behavior**: Apply ΔC in given order (no sorting, no dependencies), with conflict rule (last change overrides earlier changes to same normalized field).
- **Success Criteria**: Functional equivalence (e.g., API schema correct, tests pass; strict must-have conditions + allowed cosmetic variations), threshold ≥90% per scenario and per model.
- **Failure Detection**: Graded outcomes (success ≥90% criteria, partial 50-89%, failure <50%) based on objective metrics.
- **Evaluator**: Deterministic where possible (test cases, schema validators, unit tests); if LLM used, separate evaluator model with fixed prompt.
- **Validation Phases**: Phase 1: Test change-only ΔC sequence for continuation. If fails, Phase 2: Add minimal context (≤10 tokens, appended to each ΔC, fixed template like "Task: [task_name]"). If still fails, Phase 3: Add constraints (preconditions for ΔC, format: "requires [condition]", max 5 tokens, allowed types: field dependencies).
- **Test Scenarios**: At least 2 task types (e.g., API design, bug fix), 2 different models (e.g., Claude, GPT, one strong/one weaker or different behavior), ≥5 runs per scenario, with repeatability checks (same sequence → functionally equivalent result, 3 repetitions, variance measured as functional difference), robustness test (reorder/remove ΔC: critical remove → must fail, independent reorder → should pass), field coupling test (dependent fields scenario: coupling should break ΔC as acceptable limitation), prompt variability (fixed baseline + 1-2 variations), sequence lengths (short 3-5, medium 5-10, longer 10+), drift check (checkpoint evaluation after 3rd, 5th, final ΔC), include one messy task.
- **Baseline Comparison**: Run same tasks with full context vs ΔC to distinguish primitive failure from task complexity.
- **What Becomes Simpler (If Proven)**: No context rebuilding, cross-model handoffs, persistent workflows.

## Steps
0. **Prove ΔC Primitive (Only Focus)**: Use constrained ΔC (pattern: "change [field] to [value]"), state (list of changes), replay (apply in order with last wins for conflicts). Test with objective metrics (e.g., code compiles/passes tests, API returns expected schema). Scenarios: 2 task types, 2 models, multiple runs. Run validation phases: Phase 1 (change-only) → Phase 2 (add context) → Phase 3 (add constraints). If proven at any phase, proceed; if not, redesign.

1. **Deferred: Full System Build**: Once primitive proven, add extraction, linking, transfer, validation, MCP integration, documentation.
